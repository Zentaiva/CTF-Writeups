01 Initial Recon

The challenge description hints at a hidden admin dashboard on an old portal for BrickWorks Co. Navigating to the instance, we're greeted with a generic maintenance page — nothing obviously useful on the surface. 

http://brick-by-brick.web.ctf.umasscybersec.org:32769/
→ "Internal systems are currently undergoing maintenance."

No forms, no links, no JS calls to interesting endpoints. Time to look at what the server is deliberately hiding.

02 robots.txt — Exposed Internal Paths

The first thing to check on any web challenge is /robots.txt. It's meant to instruct crawlers which paths to skip — which conveniently tells us exactly where to look. 

robots.txtUser-agent: *
Disallow: /internal-docs/assembly-guide.txt
Disallow: /internal-docs/it-onboarding.txt
Disallow: /internal-docs/q3-report.txt

# NOTE: Maintenance in progress.
# Unauthorized crawling of /internal-docs/ is prohibited.

Three files listed. The assembly-guide.txt and q3-report.txt turned out to be dead ends — generic company content with no useful technical details. The interesting one was it-onboarding.txt.

💡
Tip: robots.txt is one of the first places to check in web CTF challenges. Developers often add sensitive internal paths there thinking "bots won't access it anyway," but the file is publicly readable by anyone.

03 LFI via ?file=

The IT onboarding document leaked two critical pieces of information:

Finding 1 — File Parameter
The internal document portal accepts a ?file= GET parameter to serve arbitrary files. No mention of any access control or path validation.
Finding 2 — Config Location
Credentials for the admin dashboard are stored in config.php located in the web root.

Combining both findings — we can request the config file directly through the exposed parameter:

urlhttp://brick-by-brick.web.ctf.umasscybersec.org:32769/?file=config.php

The server responds with the raw PHP source — completely unauthenticated:

config.php// BrickWorks Co. — Application Configuration
// WARNING: Do not expose this file publicly!

// The admin dashboard is located at /dashboard-admin.php

// Database
define('DB_HOST', 'localhost');
define('DB_NAME', 'brickworks');
define('DB_USER', 'brickworks_app');
define('DB_PASS', 'BrickW0rks_db_2024!');

// WARNING: SYSTEM IS CURRENTLY USING DEFAULT FACTORY CREDENTIALS.
// TODO: Change 'administrator' account from default password.

define('ADMIN_USER', 'administrator');
define('ADMIN_PASS', '[deleted it for safety reasons - Tom]');
⚠️
The config clearly states the system is running default factory credentials and someone named Tom removed the password from the comment but forgot to actually change it. Classic.

So we have the username: administrator. The password is blanked out in the comment, but the file itself warns that default credentials are still in use. A reasonable guess is that the default password is something generic — trying administrator as both username and password worked immediately.

04 Admin Panel — Flag Captured

We now know the admin panel is at /dashboard-admin.php. Navigating there and logging in with the default credentials gives us full access to the BrickWorks control panel — and the flag. 

urlhttp://brick-by-brick.web.ctf.umasscybersec.org:32769/dashboard-admin.php
→ Login: administrator / administrator
→ CHALLENGE COMPLETE — FLAG CAPTURED

Key takeaways:
Robots.txt is not a security mechanism — it's a roadmap. Never store credentials in source files served by the web root. The ?file= parameter had zero path validation, making it a trivial LFI. And of course — always change default credentials before deploying anything to production.
