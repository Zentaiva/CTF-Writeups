BrOWSER BOSS FIGHT

web easy 100 pts · UMass CTF 2026
challenge description

The challenge says: "This familiar brick castle is hiding something... can you break in and defeat the Koopa King?" The name BrOWSER is a hint — OWS stands for browser tricks. Let's dig into what's happening on each layer.

// vulnerabilities exploited
CLIENT-SIDEJS silently replaces the input value before form submit — bypassed by sending the POST directly
INFO DISCLOSUREPassword leaked inside a custom Server: response header
BROKEN ACCESSAuthorization relies on a client-controlled cookie (hasAxe=false)

01 Landing Page — The Castle Gate

Opening the instance we get a full-screen Bowser's castle with a single text input asking for a key. Looks intimidating, but let's check the page source first.

browser-boss-fight.web.ctf.umasscybersec.org:32769

Inside the page source there's a suspicious JS block attached to the form:

document.getElementById('key-form').onsubmit = function() {
  const knockOnDoor = document.getElementById('key');
  // It replaces whatever they typed with 'WEAK_NON_KOOPA_KNOCK'
  knockOnDoor.value = "WEAK_NON_KOOPA_KNOCK";
  return true;
};

Whatever we type gets silently swapped out for WEAK_NON_KOOPA_KNOCK before the form submits. This is purely client-side validation — the server never sees our real input. The solution: bypass the form completely and craft the POST request manually in Burp Suite.

ℹ Client-side JS can be ignored, modified, or bypassed entirely. Any validation or transformation happening in the browser is not security — it only affects users who submit through the normal UI.

02 Burp Suite — Password Hidden in Response Header

Intercepting the POST to /password-attempt in Burp and forwarding it with our own key value, the server redirects us to /kamek.html. Kamek shows up and tells us we're clearly not Bowser.

browser-boss-fight.web.ctf.umasscybersec.org:32769/kamek.html

Dead end on the page itself — but the response headers from the redirect are worth a closer look:

burp suite — response headers 
HTTP/1.1 302 Found
X-Powered-By: Express
Server: BrOWSERS CASTLE (A note outside: "King Koopa, if you forget
  the key, check under_the_doormat! - Sincerely, your faithful servant, Kamek")
Location: /kamek.html

The Server: header contains a note from Kamek to King Koopa — if he forgets the password, check under_the_doormat. That's our key.

⚠ Response headers are fully visible to anyone using Burp or browser devtools. Never put credentials, internal notes, or sensitive info in custom headers — especially Server:.

03 Getting Past the Gate

Sending the POST again — this time with key=under_the_doormat in the body instead of the JS-replaced value — the server redirects us somewhere new:

burp suite — redirect to bowsers_castle.html 
HTTP/1.1 302 Found
Location: /bowsers_castle.html

We're inside the castle now. Following the redirect:

browser-boss-fight.web.ctf.umasscybersec.org:32769/bowsers_castle.html

Bowser taunts us: "I don't know how you got in, but you can't possibly defeat me! I removed the axe!" In Mario, the axe at the end of the bridge is what defeats Bowser. The game is hinting that we need to get the axe somehow.

04 Cookie Manipulation — hasAxe=true

Intercepting the GET request to /bowsers_castle.html in Burp, there are dozens of cookies in the request — named after Mario enemies. Scrolling to the end of the cookie header, one stands out:

burp suite — request cookies 
... dry_bones_30=3740ya; fruit_named_woman_30=o3taoe; hasAxe=false

The server is using a client-controlled cookie to decide if we have the axe. Changing hasAxe=false → hasAxe=true and resending the request: 

Cookie: ... hasAxe=true

05 Flag Captured

The server responds with HTTP 200 and the flag in the response body:

// takeaways
  • Client-side JS is not validation — always intercept and inspect the actual HTTP request
  • Check every response header, not just the body — Server: headers can leak sensitive info
  • Cookies are fully client-controlled and must never be trusted for access decisions server-side
  • Always use Burp to look at the full picture: headers, cookies, redirects
